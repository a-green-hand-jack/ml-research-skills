# Memory Guidelines

Use memory to preserve project steering decisions across sessions. Save concise facts, not long reports.

## Save When Useful

Save:

- promising ideas and why they were marked `pursue`
- revised ideas and what must change before implementation
- parked ideas and the trigger for reconsideration
- killed ideas and the reason not to revisit
- closest work already checked
- user resource constraints such as compute, data, collaborators, deadlines, and codebases
- recurring reviewer risks for the user's research area
- project positioning decisions

Do not save:

- speculative brainstorm lists with no decision
- private paper content unless the user is clearly working in a local/private project context
- large copied literature summaries
- hidden reasoning

## Suggested Memory Format

```markdown
## YYYY-MM-DD - <short idea name>

- Decision: pursue / revise / park / kill
- One-sentence idea:
- Best paper claim:
- Main risk:
- Closest work to check:
- Minimum viable project:
- Kill criteria:
- Next skill/action:
```

## Reuse Memory

Before validating a related idea, check whether memory already contains:

- a similar idea that was killed or parked
- closest work already identified
- known resource constraints
- previous advisor or reviewer objections
- a project positioning decision

If memory conflicts with current user input, trust the current user input and note the change.
