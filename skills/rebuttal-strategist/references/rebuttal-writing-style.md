# Rebuttal Writing Style

Use this reference when drafting responses.

## Tone

Write:

- concise
- evidence-first
- respectful
- confident but not defensive
- specific about revisions

Avoid:

- arguing about reviewer competence
- over-apologizing
- vague promises
- "we believe" without evidence
- claiming new results without enough detail
- spending many words on low-impact comments

## Response Postures

### accept-and-fix

Use when reviewer is correct.

```text
We agree and will clarify this in Sec. X. The intended scope is [...], and we will add [...]
```

### clarify-misunderstanding

Use when text exists but was missed.

```text
We apologize for the lack of clarity. The experiment in Table X controls for [...]. We will move this explanation to the main text and state [...]
```

### rebut-with-evidence

Use when reviewer concern is contradicted by evidence.

```text
This concern is addressed by [...]. Specifically, Table X shows [...], indicating [...]
```

### partially-concede

Use when reviewer has a valid but overstated point.

```text
We agree that [...] is a limitation. However, the main claim does not require [...]; it relies on [...], which is supported by [...]
```

### provide-new-result

Use when rebuttal includes new evidence.

```text
Following the reviewer's suggestion, we added [...]. The result is [...], supporting [...]
```

### scope-and-limit

Use when the request is too broad or outside scope.

```text
We will revise the claim to make clear that it applies to [...]. Extending to [...] is important but outside the current scope because [...]
```

### defer-to-revision

Use for manuscript changes.

```text
We will add this clarification to Sec. X and include the detailed setup in the appendix.
```

## Structure

For each major issue:

1. name the concern briefly
2. answer directly
3. give evidence
4. state paper change

Example:

```text
Baseline X. We ran X under the same protocol as Table 2. Our method remains stronger on A/B by 2.1/1.7 points, while matching X on C. We will add this baseline and discuss the tie on C as a limitation.
```

## Wording Rules

- Use exact numbers when available.
- Mention table/figure/appendix references.
- Say "we will revise" only for changes the authors can make.
- Say "we added" only if the result exists.
- Use "scope" rather than "ignore" for out-of-scope requests.
- Keep reviewer-specific replies short if a unified response already answers the issue.
